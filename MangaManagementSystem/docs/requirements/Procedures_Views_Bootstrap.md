# Procedures, Views, and Bootstrap Summary

Source: `MangaManagementSystem_Procedures_Views_Bootstrap.sql`

This document summarizes stored procedures, key views, and the bootstrap routine present in the SQL script.

## Phase 7 Architecture Note

C# Application owns business decisions and orchestration for registration, authorization, Admin account management, and file-cleanup workflows. SQL Server procedures are persistence, transaction, constraint/backstop, and audit boundaries. Procedures insert, update, link records, enforce database consistency, and append audit events. Application code decides whether a request should be attempted and maps results/errors for the UI/API.

This wording is intentional for Phase 7 and matches the leader direction: **C# Application = business decision + orchestration; database = insert/update/link/audit and safety backstop.**

## Procedures

### Audit

- `audit.usp_AuditEvent_Append(actor_user_id, action_code, entity_type, entity_id, detail_json)`
  - Appends an audit event.
  - Resolves `actor_role_name` internally from `actor_user_id`.
  - Validates JSON when `detail_json` is supplied.
  - Serializes audit append with `sp_getapplock`.

### Auth / Account

- `auth.usp_User_Create(role_name, username, email, password_hash, display_name, avatar_file_id, portfolio_file_id, created_by_user_id, new_user_id OUTPUT)`
  - Creates a new user with `status_code = PENDING_APPROVAL`.
  - Resolves `role_name` to GUID `role_id`.
  - Lets `auth.Users.user_id` be generated by `DEFAULT NEWID()` and captures it through `OUTPUT inserted.user_id`.
  - Writes `USER_REGISTERED` audit event.
  - Does not create an `ACTIVE` public Admin account directly.

- `auth.usp_Admin_ChangeUserStatus(admin_user_id, target_user_id, new_status_code, reason)`
  - Updates user status through the Admin workflow.
  - Requires `admin_user_id` to be an active Admin.
  - Supports approved status transitions including pending to active/rejected/disabled, rejected to active, disabled to active, and active to disabled.
  - Requires reason for `REJECTED` and `DISABLED`.
  - Prevents Admin from rejecting/disabling their own account.
  - Writes `USER_STATUS_CHANGED` audit event.

- `auth.usp_User_UpdateDisplayName(user_id, new_display_name)`
  - Updates display name only.
  - Preserves username, email, password, role, and account status.
  - Writes display-name audit event.

- `auth.usp_User_ResetPassword(target_user_id, new_password_hash, actor_user_id, reset_mode, reset_reason)`
  - Updates password hash.
  - Preserves account approval/status state.
  - Writes password reset/change audit event.

- `auth.usp_User_UpdatePortfolioFile(...)`
  - Creates a `REGISTRATION_PORTFOLIO` file resource and links it to the user.
  - Uses GUID file IDs and audit traceability.

- `auth.usp_User_CreateWithOptionalPortfolio(...)`
  - Wraps user creation and optional portfolio linking in one SQL workflow.
  - Used after Application validates registration inputs and uploads portfolio file metadata.

- `auth.usp_User_UpdateAvatarFile(...)`
  - Creates a `USER_AVATAR` file resource and links it to the user.

### File Resource / Admin File Management

- `manga.usp_FileResource_Create(...)`
  - Creates a `manga.FileResource` row with Cloudinary metadata and SHA-256 hash.

- `manga.usp_FileResource_SoftDelete(...)`
  - Marks file metadata deleted without physically deleting the Cloudinary asset.

- `manga.usp_Admin_FileResource_Search(...)`
  - Searches file resources for Admin File Management.

- `manga.usp_Admin_FileResource_GetById(...)`
  - Reads file detail for Admin File Management.

- `manga.usp_FileResource_UpdateStorageCleanupResult(...)`
  - Persists Cloudinary cleanup result after Application performs cleanup orchestration.

- `manga.usp_FileResource_GetStorageCleanupCandidates(...)`
  - Returns soft-deleted file candidates for Application-driven Cloudinary cleanup.

- `manga.usp_Admin_FileResource_SoftDelete(...)`
  - Admin-facing soft delete workflow for file metadata.

- `manga.usp_FileResource_FindActiveDuplicates(...)`
  - Finds active duplicate candidates by SHA-256 hash.

## Notes

- GUID IDs use `UNIQUEIDENTIFIER` in the current schema.
- Business/domain table IDs should not be documented as `INT IDENTITY`, `BIGINT IDENTITY`, or `SMALLINT IDENTITY`.
- `audit.AuditEvent.audit_event_id` may remain a numeric ledger-style audit exception.
- Application-level tests should verify direct request/direct URL authorization; SQL tests should verify stored-procedure persistence and backstop behavior.
